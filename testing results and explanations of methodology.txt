~290k single-qubit gate ops/sec in-browser (C++/WASM via Embind).

I wrote a function that, upon pressing a button, called 50k gates in a 30 gate random order. So, it would do something like "val = XGate(val) => val = SGate(val) => etc" but with 50k gates. I ran that 5 times in succession, put each time measured using performance.now() into an array, sorted that array, then took the median (middle) of that array to get the mean time it took in ms. I clicked the button 5 times to prevent any variance (so there was a total of 25 results), and the medians of the means was the result I took. My runs (after npm run preview) were

Where the first one is a JIT warm up. The median within each batch, then the mean of the four post-warmup medians, 167.4, 165.5, 175.3, 174.9, is 171.15 ms. 50000 gates/ 0.17115s is 292141.396436 gates/second.

Why only 50,000 gates? Longer runs (~1M gates, ~5s of sustained compute) showed slowdown from CPU thermal throttling; shorter runs finish before the cpu gets hot, giving stable timings. Additionally, the UI only allows a few manual gate applications per second (since the users have to click manually), so throughput is nowhere near a user constraint at current scale. What the measurement shows: near-identical dev and production timings indicate per-call cost is dominated by the JS/WASM boundary rather than the arithmetic, and the figure establishes a baseline for batch execution features (premade circuits, validation harnesses) and for multi-qubit support, where per-gate cost grows with state size.

Why test on the frontend? I cared more about what the user actually experiences, now, or later with heavier multi-gate features, than how fast the C++ could theoretically run if I timed it internally behind a single call. If the matrix math was fast but slow to cross the JS/WASM boundary, the user experience could take that hit without me knowing, and I wanted to find that out ahead of time.
